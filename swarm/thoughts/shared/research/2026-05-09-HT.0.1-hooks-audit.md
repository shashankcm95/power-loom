### Stability Patterns

- All 16 registered hook scripts in `hooks.json` declare `timeout` (3-10 seconds depending on script). Timeout discipline uniform.
- `error-critic.js` uses 8-second timeout for subprocess `spawnSync` calls (line 244). Consistent with the parent hook's 5s timeout? Hook timeout is 5s in `hooks.json` line 117; spawnSync inside the hook has 8s timeout (lines 245, 251) — the inner timeout exceeds the outer. Observation: if the outer hook timeout fires first, the spawnSync result may be lost. This may be intentional (the parent timeout terminates the whole hook process and child) but the asymmetry is worth documenting.
- `auto-store-enrichment.js` line 244 — spawnSync 8000ms timeout inside Stop hook with 10000ms timeout in `hooks.json` line 162. Inner < outer. Compliant.
- `pre-compact-save.js` line 268 — spawnSync 10000ms timeout inside PreCompact hook with 10000ms timeout in `hooks.json` line 139. Inner == outer. If the spawnSync runs to its timeout, the parent hook also hits its timeout boundary. Tight margin.
- `console-log-check.js` lines 19, 31, 37 — execSync 5000ms timeout inside Stop hook with 5000ms timeout in `hooks.json` line 151. Three sequential subprocess calls each carry the full 5s timeout; the static cumulative upper bound is 3 × 5000ms = 15s versus 5000ms parent hook timeout. Wall-clock latency in practice is a runtime-untestable-statically observation (per nova FLAG-2).

Stability observation:
- `principle-observation:hooks/scripts/console-log-check.js:19,31,37` — three sequential `execSync` calls each with 5s timeout under a 5s parent-hook timeout. Static-only observation; not a runtime measurement.
- `principle-observation:hooks/scripts/error-critic.js:244-245,250-251,256-257` — inner spawnSync timeouts (8000ms and 4000ms) under outer hook 5000ms timeout (`hooks.json:117`). Inner timeout exceeds outer in the 8000ms case.
