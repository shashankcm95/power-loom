### Atomic-write + lock-the-RMW patterns

Several scripts implement the "lock the WHOLE read-modify-write cycle" pattern surfaced post-H.3.2:

- `tree-tracker.js:48-67` — `writeTreeAtomic` (no lock, atomic tmp+rename) + `withTreeLock` (lock, 15000ms timeout for chaos-test contention) wrapping the RMW callsite at `tree-tracker.js:100,152`
- `budget-tracker.js:67-83` — `writeBudgetsAtomic` + `withBudgetLock` (15000ms) wrapping `cmdRecord` RMW at `budget-tracker.js:162-182`
- `kb-resolver.js:97-104` — `writeManifestAtomic` wraps the write inside `withLock`; `cmdScan` does load → walk → write where the load doesn't need the lock since the manifest is overwritten wholesale
- `agent-identity.js:88-93` — `writeStore` is atomic tmp+rename; `withLock` at line 86 wraps RMW at all callsites (`cmdInit`, `cmdAssign`, `cmdRecord`, `cmdPrune`, `cmdUnretire`, `cmdBreed`)
- `self-improve-store.js:152-165` — `writeAtomic` with crypto-randomized tmp suffix (H.5.3 fix for PID-collision); double-lock pattern at `cmdScan` (`COUNTERS_PATH.lock` + `PENDING_PATH.lock`, lines 232-280)
- `pattern-recorder.js:65-69` — `saveStore` atomic; lock-the-RMW via `acquireLock`/`releaseLock` from `_lib/lock.js` wrapping `cmdRecord` at line 95-194
- `spawn-recorder.js:89-94` — `appendRow` uses `appendFileSync` (single-line append; OS-atomic up to PIPE_BUF) inside `withLock`
- `prompt-pattern-store.js:124-134` — `saveStore` atomic; `withLock` wraps store mutations at `cmdStore` line 188-244

`weight-fit.js` and `quality-factors-backfill.js` are read-mostly (one-shot analysis or backfill) and use a simpler tmp+rename without lock since they are not concurrent-write paths.

### ADR-0001 fail-open compliance

ADR-0001 invariant 4 (substrate writes fail loud) explicitly applies to hooks; substrate scripts called from the CLI do not have the same fail-open contract, instead using `process.exit(1|2)` on usage errors and `process.exit(2)` on lock-acquisition failures. The pattern is consistent across the audit scope: usage errors surface with `console.error` + `process.exit(1)`; lock-acquisition timeouts surface with `console.error` + `process.exit(2)`; corrupt-state surfaces with `console.error("Refusing to advance")` + `process.exit(2)` (e.g., `kb-resolver.js:88-91`, `budget-tracker.js:57-60`, `agent-identity.js:79-81`).

`build-spawn-context.js:282-285` is the one substrate script that explicitly cites ADR-0001 fail-open: top-level error caught and stderr-logged, then `process.exit(1)`. This is consistent with the script being invoked from hooks (the H.8.4 trilogy entry point) where fail-open matters.

### Path-traversal symlink check parity (kb-resolver vs adr.js, master plan special focus)

Master plan special focus per the HT.0.2 list: verify `findDocPath` symlink-check uniformity across `kb-resolver` and `adr.js` (the master plan annotation cites H.8.7 as the parity-addition phase).

Verified. Two different defenses:

**`kb-resolver.js:118-140` — `findDocPath`** is a 23-LoC function that does:
1. `path.resolve(KB_BASE, kbId + '.md')` — lexical candidate
2. Lexical first-pass check: `if (!candidate.startsWith(baseResolved + path.sep) && candidate !== baseResolved) return null;` — refuses `..`-traversal even on missing files
3. `fs.realpathSync(candidate)` and `fs.realpathSync(baseResolved)` — canonicalize
4. Re-check: `if (!realCandidate.startsWith(realBase + path.sep) && realCandidate !== realBase) return null;` — refuses real-path-escape via symlink

**`adr.js:47-55` — listAdrFiles symlink filter** uses a different shape:
1. `fs.lstatSync(path.join(ADRS_DIR, f))` — does not follow links
2. `if (!st.isSymbolicLink()) return true;` — skips any entry that is a symlink

The two defenses are not the same shape: `kb-resolver` permits real-path-bounded symlinks (canonicalizes both sides and verifies the canonical path stays under `KB_BASE`); `adr.js` forbids all symlinks unconditionally. The `adr.js` variant is the stricter of the two.

The H.8.7 phase-tag at `adr.js:44-55` ("symlink defense parity with kb-resolver") and the H.8.7 fix-comment at `kb-resolver.js:111-117` (CS-2 hacker.ren CRIT-2) confirm the two were tightened in the same H.8.7 phase. The "parity" is conceptual (both defend against symlink-escape); the implementations are not byte-for-byte identical.
