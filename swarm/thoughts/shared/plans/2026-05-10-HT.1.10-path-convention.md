# HT.1.10 — Path-convention consolidation

## Context

Per HT.0.3 finding (`commands/build-team.md` 322 LoC + 6-responsibility split + 5 path-reference conventions across 13 files) + HT.0.4 (`02-confused-user.md` mixes hardcoded author-machine + deployed-marketplace conventions in same file) + HT.0.5a (3 hets KB docs hardcode author-machine paths). The HT.1 backlog (lines 199-209) framed this as "convention doc + 8+ site sweep" with score-4 (impact-2 / effort-2).

**Empirical pre-validation per HT.1.8/HT.1.9 dogfooded pattern surfaced drift-note 70**: the perceived inconsistencies are INTENTIONAL context-dependent conventions, not bugs. Source-only files (e.g., `swarm/hierarchical-aggregate.js` which doesn't exist at `~/.claude/swarm/...`) MUST use `~/Documents/claude-toolkit/...`. Files that exist in both source AND deployed install (e.g., `scripts/compliance-probe.sh`) can use `~/.claude/...` for runtime invocation. The 5-convention spread is the substrate's actual usage pattern; what's missing is the documentation explaining when each convention applies.

**Why sub-plan-only methodology**: convention doc authoring per established `decision-record-pattern: lightweight` precedent (HT.1.6 BACKLOG.md entry shape — first-of-3 planned across HT.1 per HT.0.9-verify FLAG-5 right-sizing). Single doc artifact; no schema change; no institutional discipline beyond what's already implicit; no fresh design surface beyond documentation framing. Matches HT.1.6 + HT.1.8 + HT.1.9 sub-plan-only precedent. Per-phase pre-approval gate would not catch HIGH severity bugs that empirical pre-validation didn't already catch.

**Why now (HT.1.10 timing)**: convention doc surfaces the implicit context-dependent path semantics before more files accumulate that drift from the convention. Sequencing-correct after HT.1.9 substrate hygiene phase.
