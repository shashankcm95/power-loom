# H.9.7 — ESLint baseline + fix-all-findings + ADR-0006 'fix-don't-suppress' discipline

## Context — three motivating forces

1. **Sibling format-discipline gap closure** (continues H.9.0/9.1/9.2/9.5 pattern). H.9.x has added markdownlint + shellcheck + jq + yaml-lint to local smoke. JavaScript is the only major content class without lint coverage. 53 substrate `.js` files run unlinted; drift accumulates silently.

2. **Architect cumulative-audit FLAG (d) at H.9.5.1**: H.9.7 was re-bucketed from `borderline gate` to MANDATORY-gate because ESLint default rules + lint config decisions = institutional discipline encoding (HT.1.6 trigger 4).

3. **User directive at H.9.7 entry**: "the approach should never be to suppress and move on. It should always be fix before it gets out of control. we need to bake this into our plugin contract so we can increase reliability." This is institutional-commitment language; codified as NEW ADR-0006.

### Critical real-bug catches (substrate-as-testing-framework reframe validated)

**`contract-verifier.js:214,216`** — Two closures inside `functionalChecks` reference variable `parsed`:

```javascript
const functionalChecks = Object.assign(Object.create(null), {
  outputContainsFrontmatter: () => parsed !== null,        // line 214 — bug
  frontmatterHasFields: (cArgs) => {
    if (!parsed) return false;                              // line 216 — bug
    return cArgs.fields.every((f) => frontmatter[f] !== undefined);
  },
```

At HT.1.2 (commit `_lib/frontmatter.js` DRY consolidation), the surrounding scope was refactored from `const parsed = parseFrontmatter(output)` (returns `null` when no frontmatter) to `const { frontmatter, body } = parseFrontmatter(output)` (always returns object). The closures retain stale `parsed` references; would throw `ReferenceError` at runtime whenever these functional checks fire.

**Why dormant — CORRECTED post-code-reviewer FLAG-1**: prior sub-plan draft claimed "only documentary contracts (14/15/16) use frontmatterHasFields." That was wrong by ~6×. Empirical grep across all 18 substrate contract JSONs shows EVERY contract (01 through 16 plus `challenger` + `engineering-task`) declares `outputContainsFrontmatter` (F1, required: true) + `frontmatterHasFields` (F2, required: true). The dormant-bug surface is the FULL 18-contract set, not 3 documentary contracts. The bug has been dormant only because the substrate is pre-production for actor verification — no actor has produced output for contract-verifier to check yet. **The first real-production run would crash via ReferenceError**, not just `/research` runs. Urgency of the fix is correspondingly higher than original framing implied.

**Fix shape** at H.9.7 implementation:

```javascript
outputContainsFrontmatter: () => Object.keys(frontmatter).length > 0,
frontmatterHasFields: (cArgs) => {
  if (Object.keys(frontmatter).length === 0) return false;
  return cArgs.fields.every((f) => frontmatter[f] !== undefined);
},
```

Validates the user's "fix before it gets out of control" reframe: lint surfaces dormant bugs that would otherwise crash in production. This is the gate paying for itself in the first invocation.

## Substrate-config-strategy decision (architect gate decides)

ESLint v9 flat config (`eslint.config.js`) requires resolving `@eslint/js` for `js.configs.recommended`. The substrate has no `package.json` or `node_modules` (intentional zero-runtime-dependency property). Three options:

### Option A — Substrate becomes npm package (institutional shift)

Add `package.json` at substrate root with `devDependencies: {eslint, @eslint/js, globals}`. Add `node_modules/` to `.gitignore`. Developers + CI run `npm install` before invoking ESLint.

- **Pro**: industry-standard; reliable; works locally + CI identically; opens door for other dev tools later (prettier, additional linters).
- **Con**: first time substrate becomes an npm package — large institutional shift; new developer-onboarding step (`npm install` before testing); `package.json` becomes a load-bearing artifact requiring maintenance.

### Option B — Inline-rolled rules in `eslint.config.js` (zero new deps)

Hand-roll the `eslint:recommended` ruleset directly in substrate's `eslint.config.js` (no `require("@eslint/js")`). About 60 rule entries. `npx --yes eslint@9` runs directly from substrate root.

- **Pro**: preserves substrate's zero-runtime-dependency property; matches H.9.0-H.9.5 `npx --yes <tool>` convention; minimum institutional surface.
- **Con**: maintenance burden — when ESLint v9.x.y bumps recommended set, manually sync inline list (ESLint major-version recommended changes ~1-2/year).

### Option C — Wrapper script + tmpdir node_modules (hybrid)

`scripts/eslint-runner.sh` provisions tmpdir with `eslint + @eslint/js + globals` via `npm install`, then invokes ESLint with substrate-relative paths. smoke Test 84 calls the wrapper.

- **Pro**: substrate stays zero-dep at rest; tmpdir is ephemeral.
- **Con**: more wrapper complexity; non-standard; harder to maintain than A or B.

### Recommendation (gate-decided: Option B with explicit maintenance commitment)

**Selected: Option B** (inline rules) — gate approved. Best fits substrate convention; preserves `npx --yes <tool>` precedent.

**Maintenance commitment (per architect FLAG-1 absorption)**: any ESLint major-version bump (v9 → v10) triggers an explicit `recommended` rule-set audit + ADR-0006 amendment if the rule set changed. Substrate authors do NOT silently bump ESLint major versions; bumps go through ADR-update + per-phase pre-approval gate. This commitment makes Option B's "manual sync at major-version bumps" load-bearing rather than aspirational. ESLint minor + patch bumps (v9.x.y) are caught by the smoke harness automatically (recommended set rarely changes within a major).

## ADR-0006 — fix-don't-suppress institutional commitment

NEW governance-tier ADR codifying user directive. Shape:

- **Title**: "Fix-don't-suppress discipline — substrate maintains 0-finding lint baseline; suppression via disable comments + rule overrides PROHIBITED"
- **Status**: accepted (per HT.1.7 + HT.1.13 + ADR-0005 same-day-acceptance convention; per-phase pre-approval gate INVOKED)
- **Tier**: governance (institutional commitment; forward-looking)
- **Invariants**:
  1. Substrate maintains 0-finding lint baseline across all installed lints (markdownlint Test 80, shellcheck Test 81, jq Test 82, yaml-lint Test 83, ESLint Test 84).
  2. Lint findings get FIXED, not suppressed. `eslint-disable`, `eslint-disable-next-line`, `eslint-disable-line`, per-file rule overrides in `eslint.config.js` are PROHIBITED in substrate `.js` files (and sibling-style suppression in other lints).
  3. Exception process: when a lint rule conflicts with substrate intent (e.g., security-detection regex needs control chars), refactor the code to express intent differently (e.g., dynamic regex construction via `String.fromCharCode`) rather than suppress. ADR-update is the escape hatch if the rule is genuinely wrong for substrate (rare).
  4. New lint additions follow H.9.7 entry pattern: empirical baseline first; fix every finding; ADR-0006 invariants apply to new lint's findings.
- **Files affected**: all substrate `.js` (53 files) + `eslint.config.js` (NEW) + tests/smoke-ht.sh Test 84 (NEW) + ADR-0001 fail-open invariants remain orthogonal.
- **Consequences**:
  - Reliability gain: lint catches real bugs early (ESLint v9 baseline already caught 2 HT.1.2 stale refs).
  - Maintenance discipline: new code requires lint-clean to merge; CI enforces.
  - Suppression-disabled means real conflicts get surfaced + refactored. Slows initial development but raises baseline quality.

### Sub-phase 3 — Smoke Test 84 + Test 84b

Per gate decision (Option B + active suppression-detection per architect FLAG-2 + code-reviewer FLAG-3 convergent):

**Test 84 (ESLint baseline)**: invoke ESLint v9 via `npx --yes -p eslint@9 -p @eslint/js@9 -p globals eslint . 2>&1`. H.9.6.2 safe-pattern (`T84_EXIT=0; T84_OUT=$(...) || T84_EXIT=$?`). ESLint native output includes file paths — per-file manifest mapping NOT needed (unlike Test 83). On FAIL: `head -30` of output shows file:line:rule context.

**Test 84b (suppression-prohibition enforcement per ADR-0006 invariant 5)**: separate smoke check. `grep -rl "eslint-disable" <substrate-js-tree>` — succeeds (exit 0) if NO matches in substrate `.js` files. ADR-0006 invariant 2 prohibits `eslint-disable` tokens; Test 84b makes this enforcement deterministic at smoke time rather than relying on code-review discipline alone. Scope: exclude `node_modules/`, `.git/`, `swarm/run-state/`. Pattern shape (safe `|| T84B_EXIT=$?`):

```bash
T84B_EXIT=0
T84B_OUT=$(cd "$SCRIPT_DIR" && grep -rl "eslint-disable" . --include="*.js" --exclude-dir=node_modules --exclude-dir=.git --exclude-dir=swarm/run-state 2>&1) || T84B_EXIT=$?
# grep returns 0 if matches found (BAD for us); 1 if no matches (GOOD); >1 on error
if [ $T84B_EXIT -eq 1 ]; then
  echo "OK (0 eslint-disable suppressions in substrate .js per ADR-0006 invariant 2)"
  passed=$((passed + 1))
elif [ $T84B_EXIT -eq 0 ]; then
  echo "FAIL: eslint-disable suppressions found in substrate .js (ADR-0006 invariant 2 violated):"
  echo "$T84B_OUT"
  failed=$((failed + 1))
else
  echo "FAIL: grep exited with unexpected code $T84B_EXIT"
  failed=$((failed + 1))
fi
```
