## Phases

### Phase 1: Extract `_lib/frontmatter.js` (closes H4)

**Description**: kb-resolver.js (lines 72-87) and adr.js (lines 35-78) each have their own `parseFrontmatter` implementation with divergent bugs. Extract a shared helper.

**Files affected**:
- NEW: `scripts/agent-team/_lib/frontmatter.js` (canonical implementation)
- MODIFY: `scripts/agent-team/kb-resolver.js` (consume helper; remove inline)
- MODIFY: `scripts/agent-team/adr.js` (consume helper; remove inline)

**Approach**: Author canonical parser supporting:
- Scalar fields (string values, with optional quote-stripping)
- Inline arrays (`[a, b, c]`)
- Block-list fields (`key:` followed by `  - item` lines)
- Quoted multi-word values
- Keys with digits and underscores (per chaos eli LOW-3)
- Null marker (`null` literal) → JS `null`

Both consumers swap to `require('./_lib/frontmatter')`.

**Success criteria**:
- `_lib/frontmatter.js` exists and exports `parseFrontmatter(text)` returning `{frontmatter, body}`
- kb-resolver.js no longer has inline `parseFrontmatter`
- adr.js no longer has inline `parseFrontmatter`
- All existing kb-resolver + adr.js tests still pass (52-54 + 47-51 + 41-43)

**Manual verification**: `node scripts/agent-team/kb-resolver.js list` returns same output before/after; `node scripts/agent-team/adr.js list` same.

- [x] Phase 1 complete

### Phase 2: Fix `extractSections` fence-blindness + start-name precision (closes H1, M2)

**Description**: `extractSections` in kb-resolver.js (lines 181-196) treats `^## ` lines as section boundaries even when they appear inside fenced code blocks (` ``` `) — a kb-doc with markdown-in-code-blocks gets falsely truncated. Additionally, the start-name regex uses `\b` but matches by prefix — `## Summary` matches `## Summary of Findings` heading.

**Files affected**:
- MODIFY: `scripts/agent-team/kb-resolver.js` (extractSections function)

**Approach**:
1. Track in/out of fenced code blocks via a `inFence` boolean that toggles on lines starting with ` ``` `
2. Skip `## ` boundary detection inside fences
3. Tighten start-name match: end-of-line or whitespace-then-end after the section name (not prefix)

**Success criteria**:
- Fixture: a kb doc with `## Summary` followed by ` ``` ` then `## NotASection` then ` ``` ` then `## Quick Reference` returns Summary content stopping at Quick Reference (NOT at NotASection)
- `## Summary` does NOT match `## Summary of Findings`

**Manual verification**: Re-run install.sh tests 41-43 (kb-resolver tier tests); regression-induce by adding a test fixture with fenced `## Heading` and verify extraction.

- [x] Phase 2 complete

### Phase 3: Fix `adr.js touched-by` path-segment-aware matching (closes H2)

**Description**: adr.js touched-by (lines 245-253) uses `endsWith` for suffix matching; `barfoo.js` matches `foo.js` ADR entry; `passwd` matches `/etc/passwd`. Path-segment boundary required.

**Files affected**:
- MODIFY: `scripts/agent-team/adr.js` (cmdTouchedBy function)

**Approach**: Replace `file.endsWith(p)` with `file === p || file.endsWith('/' + p)` only (drop the bare `endsWith` clause). Same for the inverse direction. `path.sep`-aware on Windows but POSIX sufficient for current scope.

**Success criteria**:
- Fixture: ADR entry `hooks/scripts/foo.js`. Query for `hooks/scripts/foo.js` → match. Query for `hooks/scripts/barfoo.js` → no match. Query for `barfoo.js` → no match.

**Manual verification**: Re-run install.sh tests 47-51 (adr.js tests); add a regression-induction test with the false-positive case.

- [x] Phase 3 complete

### Phase 4: Fix `cmdNew` YAML escaping + ID race (closes H3, H5)

**Description**:
- H3: `cmdNew` (line 172) interpolates title directly into a `title: "..."` field; titles containing `"` corrupt YAML.
- H5: ID auto-increment (lines 145-153) is read-then-write without atomic claim; concurrent `new` invocations claim the same ID.

**Files affected**:
- MODIFY: `scripts/agent-team/adr.js` (cmdNew function)

**Approach**:
- H3: Escape `"` → `\"` (and `\` → `\\`) in title before interpolation. Also reject titles containing newlines (return error; signal user to use single-line titles).
- H5: Wrap the ID-claim + file-write in a filesystem lock (use existing `_lib/lock.js` `withLock` pattern; same as kb-resolver's manifest lock). Lock path: `${ADRS_DIR}/.cmdNew.lock`.

**Success criteria**:
- H3: Fixture: `--title 'Test "quoted" title'` produces frontmatter with `title: "Test \"quoted\" title"`; ADR loads cleanly.
- H5: Fixture: 3 concurrent `node adr.js new --title "test concurrent X"` invocations produce 3 distinct ADR IDs (no duplicates).

**Manual verification**: Manual concurrent invocation test on the H5 race fix.

- [x] Phase 4 complete

### Phase 5: Fix adr.js symlink defense + isActive null comparison (closes M3, L1)

**Description**:
- M3: `adr.js listAdrFiles` (lines 84-89) uses `fs.readdirSync` without symlink awareness; symlinks pointing outside ADRS_DIR get included.
- L1: `isActive` (line 106) checks `superseded === 'null'` (string) — but parseFrontmatter (current) parses `null` literal to JS `null`. After Phase 1's shared parser fix, the string-vs-null comparison is wrong: should compare to `null` only.

**Files affected**:
- MODIFY: `scripts/agent-team/adr.js` (listAdrFiles, isActive)

**Approach**:
- M3: After Phase 1's frontmatter extraction, add symlink check via `fs.lstatSync` + skip if `isSymbolicLink()` (or follow + verify boundary like kb-resolver's `findDocPath`).
- L1: Change `superseded === 'null'` to `superseded === null || superseded === 'null'` (defensive) — covers both old string-form and new null-form.

**Success criteria**:
- M3: Symlink in ADRS_DIR is silently ignored (not loaded as an ADR).
- L1: ADR with `superseded_by:` (no value) → null → isActive returns `true` (when status: accepted).

**Manual verification**: Add a symlink to `swarm/adrs/` pointing outside; `adr.js list` should not include it.

- [x] Phase 5 complete
