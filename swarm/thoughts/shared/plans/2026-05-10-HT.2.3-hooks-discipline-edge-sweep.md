## Part A — `_lib/lock.js` lockfile-parent-dir-missing fix (drift-note 75)

### Option A1 — Fail-fast ENOENT

**Implementation**: detect missing parent dir at acquireLock entry; throw clear error immediately.

```javascript
function acquireLock(lockPath, opts) {
  const parentDir = path.dirname(lockPath);
  if (!fs.existsSync(parentDir)) {
    throw new Error(`Lock parent directory does not exist: ${parentDir}`);
  }
  // ... existing wait loop unchanged
}
```

**Pros**:
- Makes ENOENT visible immediately; clear diagnostic for operator
- Aligns with "fail clearly" principle for substrate code
- Catches substrate-setup bugs aggressively (parent dir SHOULD exist for production callers)

**Cons**:
- Breaks the contract — consumers expecting "lock acquisition fails with timeout" now see ENOENT throw instead
- adds `fs.existsSync` + `path.dirname` overhead on every acquire (sub-millisecond; minor)
- Test fixtures still need to pre-create parent dir (gap not closed; same problem from a different angle)

### Option A2 — Auto-mkdir parent (RECOMMENDED)

**Implementation**: pre-create parent dir at acquireLock entry; transparent to consumers.

```javascript
const path = require('path');

function acquireLock(lockPath, opts) {
  fs.mkdirSync(path.dirname(lockPath), { recursive: true });
  // ... existing wait loop unchanged
}
```

**Pros**:
- Matches substrate convention (`session-end-nudge.js` lines 62 + 125 already do this for STATE_DIR)
- No consumer breakage (recursive mkdir is idempotent — no-op if dir exists)
- Closes HT.1.14 test fixture gap transparently — test 77 + future ephemeral-tmpdir tests "just work"
- Resolves drift-note 75 by ELIMINATING the failure mode (not just renaming it)

**Cons**:
- Adds `fs.mkdirSync({ recursive: true })` on every acquire (sub-millisecond cost; only writes on cold-path when dir absent; idempotent fast-path when present)
- Slightly silent (doesn't surface substrate-config bugs where parent dir is "unexpectedly" missing) — but in practice, missing parent dir IS the substrate-config bug; auto-mkdir is the fix, not a band-aid

### Recommendation: Option A2

Rationale:
1. **Substrate convention alignment**: `session-end-nudge.js` lazy-mkdir pattern is the dominant precedent; consistency reduces cognitive load
2. **Zero consumer breakage**: A1 risks breaking consumers that benefited from accidental tolerance (no evidence such consumers exist, but absence-of-evidence ≠ evidence-of-absence)
3. **Test fixture closure**: HT.1.14 test 77's manual pre-create disappears; future tests "just work"
4. **Cost negligible**: idempotent mkdir takes <0.1ms in hot path (parent exists); only writes on cold path

## Part B — `session-end-nudge.js` migration (drift-note 67)

### Option B1 — Extend `withLock` with `onLockTimeout` callback

**Implementation**: add `onLockTimeout` opt to `withLock`; hook passes custom timeout handler.

```javascript
// _lib/lock.js
function withLock(lockPath, fn, opts) {
  if (!acquireLock(lockPath, opts)) {
    const onTimeout = opts && opts.onLockTimeout;
    if (typeof onTimeout === 'function') {
      return onTimeout({ lockPath, maxWaitMs: (opts && opts.maxWaitMs) || 3000 });
    }
    console.error(`Could not acquire lock at ${lockPath} within ${(opts && opts.maxWaitMs) || 3000}ms. Aborting.`);
    process.exit(2);
  }
  try { return fn(); } finally { releaseLock(lockPath); }
}

// session-end-nudge.js
const { withLock } = require('../../scripts/agent-team/_lib/lock');

withLock(LOCK_FILE, () => {
  // load → mutate → save → write stdout
}, {
  maxWaitMs: LOCK_TIMEOUT_MS,
  onLockTimeout: () => {
    log('lock_timeout', { timeout_ms: LOCK_TIMEOUT_MS });
    process.stdout.write(input);
  },
});
```

**Pros**:
- Absorbs hook fail-soft pattern into the `_lib/lock.js` API contract; future hook callers can migrate cleanly
- Hook code becomes denser; no explicit if/return at call site
- ADR-0001 + ADR-0003 fail-soft invariant becomes parameterizable

**Cons**:
- API surface change to `_lib/lock.js` (new opt); minor but real
- All 8 existing consumers must be reviewed to confirm they don't ALSO need fail-soft (they don't — substrate-script semantics work; HT.1.8 verified)
- The hook's `input` variable is closed over from outer scope; opaque from inside `onLockTimeout`
- Introduces an asymmetry: timeout handler runs OUTSIDE the try/finally → no releaseLock attempted, which is correct (we never acquired); but conceptually muddles "withLock" semantics

### Option B2 — Use `acquireLock` + `releaseLock` primitives directly (RECOMMENDED)

**Implementation**: migrate inline acquireLock/releaseLock to `_lib/lock.js`'s primitives; preserve hook's explicit if/return.

```javascript
// session-end-nudge.js
const { acquireLock, releaseLock } = require('../../scripts/agent-team/_lib/lock');

// (remove lines 27-95 inline primitive)

process.stdin.on('end', () => {
  const haveLock = acquireLock(LOCK_FILE, { maxWaitMs: LOCK_TIMEOUT_MS });
  if (!haveLock) {
    log('lock_timeout', { timeout_ms: LOCK_TIMEOUT_MS });
    process.stdout.write(input);
    return;
  }
  try {
    // ... load → mutate → save → write stdout
  } finally {
    releaseLock(LOCK_FILE);
  }
});
```

**Pros**:
- Smaller API change to `_lib/lock.js` (none — existing primitives sufficient)
- Cleaner mapping of hook fail-soft logic at call site (explicit if/return; no callback indirection)
- PID-based stale-recovery from `_lib/lock.js` **strictly better for the crashed-holder case** (mtime+10s exists to recover from; PID-probe reclaims instantly); residual PID-rollover false-negative race is bounded by Stop hook hold-time (sub-second) × pid_max (millions on modern Linux; `kernel.pid_max` default 4194304 on 64-bit Linux) → empirically negligible on single-machine substrate (per architect MEDIUM-A2 absorption — softened from "strictly better")
- Closes drift-note 67 with minimal new surface
- Drops `sleepMs` Atomics.wait helper (lines 38-47) — code-reviewer LOW-CR2 verified `grep -n "sleepMs"` returns only 2 sites (line 38 function definition + line 79 call inside inline acquireLock); both within delete range
- Accepts `_lib/lock.js`'s busy-wait — contention bounded by Stop hook frequency × subagent count; **future-revisit trigger** (per architect MEDIUM-A1 absorption): if profiling surfaces ≥1% wall-time spent in `_lib/lock.js` busy-wait under typical HETS flows, migrate to Atomics.wait

**Cons**:
- Loses `Atomics.wait` true-sleep optimization — bounded by `maxWaitMs` (2 sec) per acquire attempt; only spins on contention; deferred per measurable trigger above
- Loses mtime-based 10s stale floor — replaced by PID-based instant reclamation (which is BETTER for the crashed-holder case; PID-rollover residual race acknowledged)
- error-critic.js sibling fail-soft-contract-upgrade still pending — per architect MEDIUM-A4 absorption: error-critic.js is already on the `_lib/lock.js` consumer surface (uses `withLock` directly with no-op fallback wrapper at lines 66-68); the deferred work is a **fail-soft contract upgrade** (migrate from `withLock` to `acquireLock`/`releaseLock` primitives so it doesn't exit-2 on timeout), NOT inline-primitive migration; smaller scope; deferred to HT.3 candidate
- **First cross-tree relative require in `hooks/scripts/`** (per code-reviewer MEDIUM-CR3 absorption): `require('../../scripts/agent-team/_lib/lock')` traverses `hooks/scripts/` → `claude-toolkit/` → `scripts/agent-team/_lib/`. Empirically verified to resolve (`node -e "require('/Users/.../hooks/scripts/../../scripts/agent-team/_lib/lock.js')"` succeeds with clean load). Residual risk: if substrate restructures hooks/ tree, the relative path breaks silently. Mitigation: install.sh test 78 implicitly covers require-resolution by invoking the Stop hook end-to-end.

### Recommendation: Option B2

Rationale:
1. **API stability**: Option B1's `onLockTimeout` opt adds API surface for one consumer; YAGNI says wait for a second consumer before generalizing
2. **Hook contract clarity**: explicit if/return at call site makes the fail-soft contract visible in session-end-nudge.js itself (which is its own institutional concern per ADR-0001)
3. **Strictly-better stale detection**: PID-based replaces 10s mtime floor with instant reclamation
4. **Drop-in scope**: existing `_lib/lock.js` primitives sufficient; no new API surface
5. **CPU cost negligible**: Stop hook contention is rare; busy-wait runs sub-100ms-per-retry only on contention
