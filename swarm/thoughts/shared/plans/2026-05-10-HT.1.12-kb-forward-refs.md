## Annotation approach decision

**Chosen**: inline YAML comment after each broken ref in frontmatter `related:` list.

```yaml
related:
  - architecture/ai-systems/agent-design  # planned — not yet authored
  - architecture/crosscut/single-responsibility
```

**Rejected alternatives**:
- **Body-section migration** (precedent-faithful to react-essentials.md): would remove broken refs from `related:` and add to body section "## Related KB docs (planned)". Rejected because: (a) react-essentials.md has NO frontmatter `related:` field — different starting point; architecture KBs HAVE existing `related:` lists. (b) Removing refs from frontmatter loses graph-walk semantics for any future bidirectional validator extension to KB docs. (c) Inline annotation surfaces the deferred-author-intent at point-of-use without structural change.
- **New `related_planned:` frontmatter field**: schema split would require updating frontmatter parsing convention + downstream consumers. Out-of-scope per HT-state.md cutover discipline (no schema change at HT.1.12).
- **Prune the broken refs entirely** (HT.1.12 backlog spec option b): would lose authorial intent (these refs were placed deliberately as forward references to planned KBs). Rejected because deferred-author-intent has documentary value per react-essentials.md precedent.

**Inline YAML comment is**:
- Minimal-invasive (1-line change per ref; no structural schema mutation)
- YAML-parser-transparent (`parseFrontmatter` + downstream parsers ignore comments)
- Human-visible at point-of-use (annotation co-locates with the ref)
- Precedent-faithful in spirit (annotation visible per react-essentials.md design intent; just located in frontmatter rather than body section to match this file class's existing structure)
