# Hardening Track — Master Plan

## Goal

By the end of the Hardening Track, the power-loom codebase satisfies five user-stated quality bars:

1. **No dead code.** Every function, exported helper, script, hook, persona, contract, KB doc, and pattern has at least one caller / reference / consumer / use site (or is a documented entry point). Code "just sitting there" is removed or restored to active.
2. **Architectural compliance.** Code measures cleanly against the 10 KB architecture docs we wrote (single-responsibility, dependency-rule, deep-modules, idempotency, acyclic-dependencies, error-handling-discipline, trade-off-articulation, reliability-scalability-maintainability, stability-patterns, rag-anchoring) + the four-dimensional context engineering doctrine adopted in H.8.6. Genuine trade-offs are documented as ADRs; mistakes are fixed.
3. **No bloat.** Where simpler/cheaper alternatives achieve the same outcome, we apply them. Over-engineered abstractions are retired. Speculative generality (YAGNI violations) is removed.
4. **Well-optimized.** Runtime cost is named, measured, and within budget. Subprocess fan-out, hook latency, and KB-injection token cost are bounded with explicit thresholds.
5. **Gap analysis surfaces all "stated vs actual" drift.** Every claim made in CHANGELOG / SKILL.md / Convention G / pattern docs / persona contracts / ADRs is verified against the code. Drift becomes either a code fix (code wrong) or a doc fix (doc wrong).

**Final docs sweep is consolidated to ONE phase (HT.2) per user instruction** — including README. Per-phase doc updates are deferred to that phase.

## Audit slicing strategy

The codebase has multiple intersecting concerns. Two slicing options:

- **Horizontal (by area type)** — audit hooks, then substrate scripts, then commands, then personas, etc. Fewer entanglements per phase; each phase's research artifact is self-contained.
- **Vertical (by feature)** — audit the prompt-enrichment loop end-to-end, then the route-decision loop, etc. Better narrative coherence; requires every phase to touch every layer.

**Choice: horizontal.** Each layer has different invariants and different stakeholders (ADRs apply to hooks; SOLID applies to substrate scripts; bidirectional `related:` applies to pattern docs). Horizontal slices match the discipline boundaries.

### Synthesis-then-refactor sequencing trade-off (per architect FLAG-4)

The plan defers all refactors to HT.0.9 synthesis → HT.1.N execution rather than fix-as-you-find during each audit phase. This choice has a real cost: each refactor phase re-opens files the audit phase already had loaded — ~15× the file-load tax across the 15-item backlog. We accept that cost in exchange for three benefits:

1. **Cross-layer dedup at HT.0.9** — a `parseFrontmatter`-class issue surfaces in both HT.0.1 (hooks) and HT.0.2 (scripts); fix-as-you-find produces two redundant fixes; synthesis catches the dedup at one decision point.
2. **ADR-vs-code categorization at one decision point** — some "violations" are intentional trade-offs that should be codified as ADRs rather than fixed. Synthesis is where that boundary call happens; per-phase fixers don't have cross-phase context.
3. **Protected documentary discipline in audit phases** — fix-as-you-find tempts critique-leak into research artifacts (the audit author becomes the fixer; the artifact becomes prescriptive). Deferring fixes to HT.1.N preserves the documentary contract (per H.8.6 14/15/16 persona contract lineage).

The trade-off is correct; the file-load tax is real. Accept-and-document.

## Per-audit-phase methodology (applies to HT.0.1 – HT.0.8)

Each audit phase follows the same shape:

### Step 1 — Inventory

Enumerate the files in scope (paths + LoC + last-modified phase tag). Output: a list of every file the audit covers. Why: prevents missing files; produces a traceable "I looked at all of this" artifact.

### Step 2 — Apply the 5 quality bars

For each file in scope, walk through:

#### A. Dead code check (mechanical 3-step procedure per nova FLAG-1)

**Procedure** (run for each file in scope):

1. **Function-level reachability**: for each `function X(...)` and each `module.exports.X = ...`:
   - `grep -rn "\bX\b" --include='*.js' --include='*.md' --include='*.json' --include='*.sh' .`
   - Count matches OUTSIDE the file itself. Zero matches OUTSIDE = candidate dead code.
2. **Script-level invocation**: for each top-level script (`scripts/agent-team/*.js`, `hooks/scripts/*.js`):
   - `grep -rn "<script-basename>" --include='*.json' --include='*.md' --include='*.sh' .` (hooks.json / commands / install.sh / other scripts)
   - Plus check `package.json` if relevant. Plus `.github/workflows/*.yml`.
3. **Dynamic-require carve-out**: greppable false positives include dynamic `require()` (e.g., `require(path.join(dir, name))`). Note these as "dynamic-load suspects, not dead" — exempt from finding.

**Surface as** `dead-code:<file:line>` with rationale (zero callers AND not dynamic-load suspect).

#### B. Architectural compliance check
Apply the 10 KB pattern docs as yardsticks:
- **Single Responsibility** — does each file have one job? Functions <50 lines?
- **Dependency Rule** — does outer code depend on inner abstractions, not the other way?
- **Deep Modules** — narrow interfaces? Hidden complexity?
- **Idempotency** — repeated invocations safe?
- **Acyclic Dependencies** — module graph DAG-shaped?
- **Error Handling Discipline** — fail-open per ADR-0001 where applicable; fail-closed for security gates; consistent `logger('error', ...)` discipline
- **Trade-off Articulation** — non-obvious choices documented as ADRs or `_NOTES.md` entries
- **Reliability/Scalability/Maintainability** — appropriate for hook-scale (low) vs distributed-scale (n/a)
- **Stability Patterns** — timeouts on subprocess; circuit-breakers where recurrent failure justifies; retry-with-backoff where applicable
- **RAG Anchoring** — kb-resolver tier-aware loading correctly used for KB injection?

Surface as `principle-violation:<principle>` with file:line.

#### C. Bloat check
- Files >800 LoC flagged for split candidate
- Functions >50 LoC flagged for extraction candidate
- Speculative abstractions (e.g., classes with 1 instance, abstract base with 1 concrete) flagged
- Convenience functions used in 1 site flagged for inline candidate
- Subprocess composition where in-process require would suffice

Surface as `bloat:<type>` with rationale.

#### D. Optimization check (static-analysis-only methodology per nova FLAG-5)

Live timing requires running hooks against real sessions; out of scope for static audit. Static-only signals:

- **Subprocess count per script**: `grep -cn "execSync\|execFileSync\|spawn\|fork" <file>`. >2 in a single script = candidate latency hotspot.
- **Synchronous fs reads in hot paths**: `grep -n "readFileSync\|statSync\|existsSync" <file>` — flag if used inside event-driven entry points (process.stdin handlers, exported request-handlers); fine if at module load.
- **Regex compilation per call vs at module load**: regex literals declared inside functions (recompiled on each call) vs at module top. Flag the inside-function ones for module-load promotion candidates.
- **KB-injection tier**: kb-resolver `cat` (Tier 3) callers checked — Tier 1 (`cat-summary`) sufficient where injection is exploratory; Tier 2 (`cat-quick-ref`) for design refresher; Tier 3 only when full doc body needed. Flag mismatches.

**Live timing benchmark** (cumulative subprocess ms per HT.X.Y session, per-hook p50/p99 latency) is a **named gap** for post-HT empirical work, NOT in audit scope.

**Surface as** `optimization:<class>` with file:line + static measurement (count, not ms).

#### E. Gap analysis (algorithm definition per nova FLAG-2)

**Source priority** (when multiple stated-behavior sources exist):
1. CHANGELOG.md (canonical user-facing claim)
2. JSDoc / inline comments at the function (canonical implementer-side claim)
3. SKILL.md phase entries (canonical phase-history claim)
4. Pattern docs (canonical doctrine claim)
5. README / repo docs (out of scope here; HT.2 covers)

When sources disagree → flag as `gap:source-disagreement` with both sides.

**Drift threshold** (what counts as "stated vs actual" drift):
- **Drift** = source claims behavior X; static read of code shows behavior Y where X ≠ Y. Flag.
- **Drift** = source claims existence (e.g., "this hook fires on Edit|Write"); code does not implement. Flag.
- **NOT drift** = source under-documents (code does X+Y; source mentions only X). Flag as `gap:under-documented` (lower severity).
- **Determination is static-only**: read the code; no runtime execution. Runtime-dependent behavior (e.g., "hook completes in <500ms typical") is NOT audited here — flag separately as `gap:runtime-untestable-statically`.

**Surface as** `gap:<type>` with both stated source (file:line + claim) and actual code (file:line + behavior).

### Step 3 — Documentation discipline (per RPI documentary contract 14/15/16 lineage)

The audit research artifact is **DOCUMENTARY**, not prescriptive:
- Describes findings; does not propose fixes
- File:line citation for every claim
- Zero "we should" / "this is bad" / "needs refactoring" language in the artifact
- Verdicts captured as observations: "function X has Y callers" not "X is dead code, remove it"
- Synthesis (HT.0.9) is where critique synthesis happens; the audit is research

This is exactly the discipline H.8.6 codified for the 14/15/16 documentary personas. Audit phases dogfood it at the human level.

**Mechanical enforcement gate (per architect FLAG-3 + nova FLAG-3 — convergent; refined per v3.1 ari NEW-1 + jade FLAG-3)**: after authoring the research artifact, run a forbidden-phrase grep BEFORE marking the phase complete:

```bash
grep -inE "\bshould be\b|\bwe should\b|\bshould\b|\bneeds refactoring\b|\brecommend\b|\banti-pattern\b|\bfix this\b|\bis bad\b|\bsmell\b|\bbetter approach\b|\bshould fix\b" \
  swarm/thoughts/shared/research/2026-05-XX-HT.0.N-<area>-audit.md
```

The `-i` flag makes matching case-insensitive (catches `FIX THIS` etc.). Word-boundaries (`\b`) prevent matching mid-word (e.g., `boulder` vs `should`).

**False-positive carve-out**: legitimate descriptive prose may use these phrases in three forms — (a) inside backtick-wrapped code citations (`` `anti-pattern` ``), (b) inside markdown blockquotes citing prior reviewers, or (c) as factual reference to existing artifacts (e.g., "section A4 of the antiPattern catalog"). After running the grep, **review each match manually**. Mark `[x]` only if all matches are documented as carve-outs. Any unhandled prescriptive phrasing → revise to documentary form (e.g., "function X has Y callers" instead of "X should be removed").

This is the same antiPattern A4 the 14/15/16 contracts run on documentary-persona output — lifted to the root-level workflow because root-authored artifacts don't pass through `contract-verifier.js`.

### Step 3.5 — Intra-phase progress marker (per nova FLAG-4)

Each research artifact opens with a `## Progress marker` section listing files inventoried so far during authoring. Updated as files are processed. Mid-session crash recovery: next session reads the artifact, sees which files were inventoried, picks up at the first un-inventoried file. Mid-session commit cadence: `chore(ht):` prefix at natural sub-breaks (e.g., after each major sub-category of files within a phase) preserves intra-phase state in git history.

### Step 4 — Output to thoughts/shared/research/

**Citation format (per jade v3.1 mechanical-gap)**: all file:line citations in research artifacts use **repo-root-relative paths**, not absolute. Example: cite `scripts/agent-team/kb-resolver.js:181`, NOT `/Users/.../scripts/agent-team/kb-resolver.js:181`. This makes findings comparable across phases and lets HT.0.9 dedup cross-layer citations of the same file by string match.

Each audit phase writes its artifact to `swarm/thoughts/shared/research/2026-05-XX-HT.0.N-<area>-audit.md` with required frontmatter (per `swarm/thoughts/shared/research/README.md`):

```yaml
---
date: ...
researcher: <persona.identity or "root">
git_commit: ...
branch: ...
repository: power-loom
topic: "HT.0.N — <area> audit"
tags: [research, hardening-track, <area>]
status: complete
last_updated: ...
last_updated_by: ...
---
```

Body sections per phase:
1. **Inventory** — files in scope (paths, LoC, last-modified)
2. **Findings — dead code** (per file)
3. **Findings — architectural compliance** (per principle, per file)
4. **Findings — bloat** (per file)
5. **Findings — optimization** (per file)
6. **Findings — gap analysis** (stated vs actual)
7. **Follow-up questions for plan phase** (anything that surfaced as critique-territory but didn't belong in the documentary output)

## Refactor phases (HT.1.1 – HT.1.N)

Each refactor item ships as a separate HT.1.N phase. Each phase:

1. Creates a sub-plan at `swarm/thoughts/shared/plans/2026-05-XX-HT.1.N-<item>.md`
2. Optionally runs pre-approval verification if substantive (multi-file or architectural)
3. Executes via `/implement` workflow (phase-by-phase pause-for-verification)
4. Updates the master HT.1 backlog with `[x]` on the item
5. **Captures pending doc-updates in a `## Doc updates pending HT.2` section of the sub-plan** (per architect FLAG-5)
6. Commits + ships as a separate PR (low-risk, well-bounded per item)

**Doc-update queue mechanism (per architect FLAG-5; refined per v3.1 ari NEW-2 + jade #3)**: each HT.1.N sub-plan has a mandatory `## Doc updates pending HT.2` heading where the refactor author lists doc edits this refactor implies. **Item shape (canonical)**: each line is `- <repo-relative-doc-path>: <one-sentence edit description>`. Example:

```
## Doc updates pending HT.2
- CHANGELOG.md: Add HT.1.4 entry noting parseFrontmatter dedup (one paragraph)
- skills/agent-team/patterns/forcing-instruction-family.md: Bump active count 10 → 11 (Convention G table)
- skills/agent-team/SKILL.md: Add HT.1.4 phase entry (3-5 sentences)
```

**HT.2 read-tax mitigation**: HT.2 reads ONLY the `## Doc updates pending HT.2` *section* of each sub-plan — not the full sub-plan body. Subject-line sections are bounded (typically <500 tokens each); 15 × 500 = ~7500 tokens for queue aggregation, well under the FIC 60% ceiling. The section MUST be **self-contained**: every doc-edit description includes file:line citations and enough context to act on without reading the surrounding sub-plan. If the description requires referring back, the section is incomplete.

Cost: 5 min per phase to author. Benefit: HT.2 doesn't re-derive doc-impact from `git log`; HT.2 stays under FIC ceiling; aggregate queue is structurally comparable across 15 sources. This is the load-bearing mechanism that lets HT.2 be a pure-docs phase without losing accuracy or blowing context budget.

**Branch discipline (per nova FLAG-8)**: HT.1.(N+1) branch is cut from main only AFTER HT.1.N PR is merged. No two HT.1.N branches open simultaneously. This avoids merge conflicts on shared files (e.g., CHANGELOG.md, SKILL.md) and keeps the master plan checkbox state consistent with shipped reality.

**No docs editing during HT.1.N phases.** Per user instruction, all doc edits consolidated to HT.2 — except the `## Doc updates pending HT.2` queue entries in sub-plans (which are *queue items*, not doc edits).

**Soak counter**: each HT.1.N runtime change resets soak. v2.0.0 timeline pushes back 4-8 weeks (acceptable per prior agreement).

## Context discipline & multi-session continuity

The Hardening Track will span many sessions. Without explicit context-discipline, two failure modes loom:
1. **Context rot** — late-phase sessions hit 60-80% context utilization, slip into the dumb zone, produce drift-laden findings
2. **State amnesia** — session N+1 can't reconstruct what session N decided / shipped / left half-done

This addition codifies a **basic-but-real** continuity discipline using the four-dimensional context engineering doctrine adopted in H.8.6 (correctness > completeness > size > trajectory) and the humanlayer "thoughts" pattern. No new tooling — just discipline + one durable artifact.

### Frequent Intentional Compaction (FIC) policy for HT

Per ace-fca.md canonical: target **40-60% context utilization** throughout. Concrete rules:

- **Hard ceiling: 60%** (Claude Code's auto-compact at 95% is way past the dumb zone). When session approaches 60%, compact intentionally before starting the next audit phase.
- **Compact triggers** (any one fires the compaction routine):
  - End of any audit phase (clean break point)
  - End of any HT.1.N refactor phase
  - Mid-phase if context crosses 60% (rare; refactor phases are bounded)
  - At explicit user `/compact` invocation
- **Compaction discipline** (preserve hints — explicit, not implicit):
  - Compact preserve list MUST include: master plan path, current phase, latest research artifact paths, current git branch, any active FLAGs from pre-approval verification
  - Per ace-fca.md "what to compact": file searches, code-flow understanding, test/build logs, JSON tool-output blobs — these are all noise after the per-phase research artifact captures the signal
- **Don't over-compact**: per Anthropic guidance, *"overly aggressive compaction can result in the loss of subtle but critical context whose importance only becomes apparent later."* Keep the latest pre-approval FLAGs + recent execution decisions verbatim; drop only the search/exploration noise.

**Lifecycle**:

- "Next concrete step" eliminates re-derivation cost — the prior session left the breadcrumb

### Per-session cutover routine (end-of-session)

When a session ends with HT work in flight:

1. **Commit any in-progress work** (research artifact / plan checkbox update / partial refactor on a branch)
2. **Update HT-state.md** with:
   - Current phase status (which `[x]` was just marked)
   - Latest artifact paths
   - Current git state (branch + sha)
   - Any open questions or in-flight FLAGs
   - Next concrete step
3. **Commit HT-state.md update** (small, fast — chore commit shape)
4. **Surface to user**: "Session ended at HT.X.Y. State saved. Next session: read `swarm/thoughts/shared/HT-state.md` before any work."

### Per-session resume routine (start-of-session)

When a new session opens with HT work pending:

1. **Read `swarm/thoughts/shared/HT-state.md` FULLY** (top priority — every session)
2. **Read the master plan** (this document) for the broader context if not in active memory
3. **Read the latest research artifact** referenced in HT-state if continuing an audit phase
4. **Read the current sub-plan** referenced in HT-state if executing a refactor phase
5. **Verify git state** matches `git_branch_at_last_save` + `git_commit_at_last_save`; if mismatch, surface to user before continuing
6. **Continue at "Next concrete step"** from the state file

This routine eats ~2-3K tokens at session start (state + master plan + relevant artifact). Acceptable tax for continuity. Below the FIC 40% lower bound, leaves headroom for the actual work.

### Milestone tracking — single source of truth

The **master plan checkbox section** (at the bottom of this document) is the canonical milestone tracker. As phases complete, mark `[x]`. The state file mirrors but doesn't replace this — if state and master plan disagree, master plan wins (it's git-tracked; state file is more frequently updated and could drift).

**Each milestone has THREE artifacts that must all exist for the milestone to be considered "complete":**
1. The phase's research artifact OR sub-plan (in `thoughts/shared/research/` or `thoughts/shared/plans/`)
2. The corresponding `[x]` mark in this master plan's checkbox section
3. (For refactor phases only) A merged PR with the code changes

If any of the three is missing, the milestone is in-flight, not complete. This catches "claimed done but not actually done" drift (the H.8.4 mio "57/57" pattern at HT scale).

### Compaction safety net

If a session crashes / context overflows mid-execution **without** the cutover routine running, the next session recovers via:

1. `git log --oneline -20` — what landed
2. `git status` + `git diff` — what was in flight
3. The master plan checkboxes — what's still `[ ]`
4. The thoughts/shared/research/ + thoughts/shared/plans/ files — what was authored

The combination is robust: even WITHOUT HT-state.md being current, the master plan + filesystem + git history give enough state to resume. HT-state.md is the optimization, not the only path.

### Why this shape is "basic but real"

- **One artifact** (HT-state.md), not five — no proliferation
- **No new tooling** — uses existing thoughts/ filesystem + git + the master plan we already authored
- **Cap-bounded** (~3-4K tokens) — read tax is finite per session
- **Three-artifact milestone rule** — catches state-vs-claim drift
- **Compaction policy is a discipline, not enforcement** — relies on user/Claude following the rule (per README's stated stance on instruction-following) but the safety net (master plan + filesystem) catches violations

This is the minimum viable continuity layer for a multi-session complex track. Heavier tooling (automated state updates via Stop hook, multi-agent handoff artifacts a la humanlayer's create_handoff command) is deferred until empirical evidence suggests v1 isn't enough.
