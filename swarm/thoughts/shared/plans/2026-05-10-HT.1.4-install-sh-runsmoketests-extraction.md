## Context

HT.0.7 audit ranked it among the 5-phase multi-responsibility-at-lifecycle-entrypoint pattern. **Sibling decision under ADR-0002**: this is the second application of the bridge-script entrypoint criterion (HT.1.3 was the first). Per ADR-0002 cross-language application, bash post-split shape is "sourced-file decomposition where the original test runner sources phase-grouped helper scripts which contribute test cases to the parent shell scope." Same criterion (lifecycle coherence + ≤800 LoC AND ≤5 responsibilities); different post-split shape from HT.1.3's Node.js dispatcher pattern.

**Why ADR-0002 fires for HT.1.4**: 1188 LoC > 800 LoC bound. Lifecycle coherence holds (single `bash install.sh --hooks --test` invocation; tests run in sequence). Responsibility count is the 4 phase eras (pre-H.x cohort + H.4.x + H.7.x + H.8.x; HT.1.x is the 5th candidate as it grows). LoC bound is the primary trigger.

## Substantive LoC discovery (capturing as drift-note 63)

**HT.0.9-verify code-reviewer FLAG-2 was incorrect**: claimed `run_smoke_tests()` spans lines 218-754 = 537 LoC. The actual function spans lines 218-1406 = **1188 LoC** (1.55× the prior corrected estimate; closer to the original draft's "~1119 LoC" pre-FLAG-2 claim). Line 754 is `}` inside the `SETTINGS_EOF` heredoc that closes at line 755 — not the function close. Subsequent function body continues through tests 39-70 + the H.8.7 trailer test 65, terminating at line 1406.

Verified by grep `^run_smoke_tests\(\) \{|EOF$|SETTINGS_EOF$` showing 14 heredoc body terminators between lines 218-1406, confirming the function continues across multiple `^}`-shaped heredoc body braces.

**Drift-note 63** captured during HT.1.4 design: HT.0.9-verify code-reviewer FLAG-2 LoC correction was itself wrong; HT.0.9-verify's "HIGH severity LoC correction" verdict needs revisiting in HT-end retrospective. The line-counting methodology (column-0 `}` matching) was unreliable for heredoc-heavy bash files. Future per-script LoC validations should use `awk` function-span detection or balance-aware brace counting, not column-0 grep. This is the second per-phase pre-approval LoC correction-of-correction across the HT.1 sequence (the first was HT.1.2's 4-vs-5 site enumeration which paid off; HT.1.3's 1698 LoC + 5 responsibilities was correct; HT.1.4's 537 → 1188 reverses HT.0.9-verify's correction).

## Implementation strategy — bash sourced-file decomposition

`run_smoke_tests()` post-extraction (~30 LoC vs 1188 pre-extraction; 97% reduction):

```bash
run_smoke_tests() {
  local passed=0
  local failed=0

  echo ""
  echo "=== Hook smoke tests ==="
  echo ""

  # Source phase-era test groups; each contributes test cases that mutate
  # parent-scope passed/failed counters via bash's lexical-scope inheritance.
  # Sourcing order preserves execution order (1-68 → 69-70 → 65 trailer).
  source "$SCRIPT_DIR/tests/smoke-h4.sh"
  source "$SCRIPT_DIR/tests/smoke-h7.sh"
  source "$SCRIPT_DIR/tests/smoke-h8.sh"
  source "$SCRIPT_DIR/tests/smoke-ht.sh"

  echo ""
  echo "  Results: $passed passed, $failed failed"
  [ "$failed" -gt 0 ] && echo "  Some tests failed — check hook scripts and paths"
}
```

Each sourced file contains the verbatim test bodies (heredocs + bash-test scaffolding) extracted from the original function. The `local passed` and `local failed` declarations stay in `run_smoke_tests()`; sourced files mutate them via parent-scope inheritance (standard bash sourced-file behavior).

`SCRIPT_DIR` is defined at install.sh:19 (`SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"`) — already a top-level constant; sourced files inherit it via parent shell scope.
