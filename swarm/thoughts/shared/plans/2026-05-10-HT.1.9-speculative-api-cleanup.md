# HT.1.9 — Speculative-API exports cleanup sweep

## Context

Per HT.0.1 D-finding (3 hooks/_lib/ modules with 7 named exports + 0 external callers) + HT.0.2 D-finding (6 substrate scripts with module.exports + 0 external callers) + HT.0.6 E-finding (`adr.js` exports {loadAllAdrs, isActive, findAdrById} with 0 external callers): the substrate has accumulated 21 speculative-API exports across 7 files. These are defensive `module.exports` blocks ("just in case someone wants to require") with no actual consumers.

Per backlog Decision (line 195): "(a) gate test-only exports behind `__test_internals__` (existing pattern in agent-identity.js); (b) delete genuinely unused; (c) accept-as-future-API only if explicitly named in BACKLOG candidates."

**Why now (HT.1.9 timing)**: speculative-API surface convergence is sequencing-correct after HT.1.8's `_lib/lock.js` adoption push (which converged the lock primitive consumers). Each new export adds maintenance cost (someone might require it; it must continue to work) without proportional value. Per YAGNI, deletion is the right shape.

**Why sub-plan-only methodology**: mechanical sweep against well-bounded empirical inventory; no fresh design surface (per-export consumer counts verified empirically; no ambiguity about which exports are speculative); no schema change; no institutional discipline encoding. Matches HT.1.2 (parseFrontmatter DRY) + HT.1.4 (mechanical bash extraction) + HT.1.6 (mechanical persona MD authoring) + HT.1.8 (`withLock` DRY consolidation) sub-plan-only precedents. Per-phase pre-approval gate would not catch HIGH severity bugs that empirical pre-validation gate didn't already catch.
