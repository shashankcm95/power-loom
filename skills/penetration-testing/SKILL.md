# Penetration Testing

Specialist skill for the `12-security-engineer` HETS persona (and any other persona that needs penetration-testing / web-vulnerability-assessment expertise). Loaded on demand via the `Skill` tool when the spawn prompt lists `penetration-testing` as required.

## When to use this skill

Trigger when:
- Auditing an application's auth flow (OAuth2, OIDC, session management, MFA)
- Reviewing input handling for injection classes (SQLi, XSS, SSRF, command injection, deserialization)
- Assessing API security (BOLA / IDOR, mass assignment, rate-limit bypass, auth bypass)
- Threat modeling a feature before it ships (STRIDE / attack-tree)
- Reviewing a security boundary's enforcement (CORS, CSP, sandbox isolation, multi-tenancy)
- Investigating a suspected vulnerability or weakness during incident response

**Skip** when the work is pure cryptography theory (use `cryptography` skill); compliance-only review (SOC2, HIPAA controls — that's audit, not pentest); secure-coding tutorials without a target system.

## Core competencies

### Methodology

- **Know your scope**: what's in-scope, what's out, what's the goal (find any issue / find specific class / verify fix). Pentest without scope produces noise.
- **Reconnaissance first**: enumerate endpoints, parameters, dependencies, deployment topology before testing anything. The map dictates the territory.
- **Threat-model before exploit**: STRIDE (Spoofing / Tampering / Repudiation / Information disclosure / Denial of service / Elevation of privilege) → identify high-value attack surfaces → prioritize.
- **Verify, don't just claim**: every finding has a reproducible PoC (request/response, code path, test case). Findings without verification are guesses.
- **Severity calibration**: CVSS 3.1 base score is the lingua franca. CRITICAL ≥ 9.0, HIGH 7.0-8.9, MEDIUM 4.0-6.9, LOW < 4.0. Don't inflate; inflated severity rots your credibility on future findings.

### OWASP Top 10 (2021) — current categories

1. **A01: Broken Access Control** — auth-z bugs (BOLA / IDOR / privilege escalation / function-level auth bypass)
2. **A02: Cryptographic Failures** — weak crypto, missing encryption-at-rest, plaintext transit, hardcoded keys
3. **A03: Injection** — SQLi, XSS, command injection, LDAP/XPath/header injection
4. **A04: Insecure Design** — flaw in design itself (e.g., password reset that emails the password)
5. **A05: Security Misconfiguration** — default credentials, verbose errors, exposed admin panels, disabled security headers
6. **A06: Vulnerable + Outdated Components** — unpatched libraries with known CVEs
7. **A07: Identification + Authentication Failures** — weak session handling, missing rate-limit on login, predictable session IDs
8. **A08: Software + Data Integrity Failures** — unsigned code execution, deserialization, supply-chain
9. **A09: Logging + Monitoring Failures** — events that should be logged but aren't (auth failures, integrity violations)
10. **A10: Server-Side Request Forgery (SSRF)** — server fetches attacker-controlled URLs without validation

### OAuth2 / OIDC specific (load-bearing for this auth audit)

- **Authorization Code Flow + PKCE is the modern default** for any client (web, mobile, SPA). Implicit Flow is deprecated. Resource Owner Password Credentials is forbidden.
- **State parameter** (CSRF defense): MUST be cryptographically random, single-use, validated on callback. Missing or static state = trivial CSRF on the auth flow.
- **Nonce parameter** (replay defense for OIDC): generated by client, returned in id_token, validated. Missing nonce = replay vulnerability for tokens.
- **Redirect URI validation**: server MUST exact-match (not pattern-match) registered redirect URIs. Pattern-match → open-redirect → token theft. (See OWASP ASVS V3.1).
- **Token storage on the client**: httpOnly + Secure + SameSite cookies > localStorage. localStorage is XSS-readable; cookies aren't.
- **Token lifetime**: short access tokens (15-60 min), longer refresh tokens with rotation. Access token stolen → bounded blast radius.
- **Token revocation**: refresh-token-rotation + token-introspection-on-each-use. Stolen-token detection via "sibling refresh" (two devices using the same refresh token = one is stolen, revoke both).
- **JWT specifics**: signature verification must precede claim parsing (`alg: none` attack); never trust claims from unverified tokens. Algorithm allowlist (only `RS256` / `ES256`); reject `HS*` if the verifier expects `RS*`.
- **Scope handling**: server-side scope enforcement at every endpoint, not client-side. Client-requested scopes are advisory; server must verify the granted scope matches the action.

### Common test patterns

**Auth bypass tests**:
- Tamper with session cookie / JWT — does the server accept tampered values?
- Try `Authorization: Bearer null` / empty string / known-other-user's token — does the server reject?
- Force-browse to admin endpoints with non-admin session — auth-z check on EVERY endpoint?
- Try API endpoints without any auth header — graceful 401 or info-leak 500?

**Injection tests**:
- Single quote in every input → SQL error message? `'-OR-'1'='1` → bypass auth?
- `<script>alert(1)</script>` in inputs that get rendered → reflected XSS? Stored XSS via reflective fields?
- `${jndi:ldap://...}` in any user-controlled input that hits a logger (Log4Shell shape)?
- Path traversal `../../etc/passwd` in file-handling endpoints?

**Token-handling tests** (OAuth2-specific):
- Replay a captured `code` from one client to another — server should reject (PKCE binds code to client)
- Replay an expired access token — server should 401, not silently accept stale claims
- Inspect `id_token` — does the server validate `iss`, `aud`, `exp`, `nbf`, `iat`, `nonce`?
- Check refresh-token rotation — does the server invalidate old refresh tokens after issuing a new one?
- Test the redirect URI — pattern-match (vulnerable) vs exact-match (correct)?

**API tests (BOLA / IDOR)**:
- Replace your `userId` param with another user's id — server returns their data? (Broken Object Level Authorization)
- Mass-assignment: POST with extra fields the schema doesn't list → server accepts and persists?
- Rate limiting at login — bypassable via header tricks (X-Forwarded-For spoofing) or distributed sources?

### Tooling

- **Burp Suite** (Community / Pro): manual + semi-automated web testing — proxy, intruder, repeater, scanner.
- **OWASP ZAP**: open-source alternative to Burp; good for CI integration.
- **sqlmap**: focused SQLi exploit + extraction.
- **nuclei**: template-driven vulnerability scanner; good for known-CVE checks.
- **ffuf / gobuster**: directory + file fuzzing.
- **JWT.io** (debugger only) + custom scripts (Python `jwt`, Node `jsonwebtoken`) for token analysis.

### Severity rationale (writing findings)

For each finding, the writeup should answer:
1. **What's the issue** (technical description)
2. **Where** (endpoint, parameter, code path with file:line if available)
3. **Reproducer** (PoC: request/response, exact command sequence)
4. **Impact** (what an attacker can do; CVSS rationale)
5. **Fix** (specific remediation, not "improve security")
6. **Validation** (how to verify the fix works)

A finding without all 6 is not actionable.

## Common pitfalls (in pentest writeups)

1. **Inflated severity** — labeling every CSRF as CRITICAL. Reserve CRITICAL for issues with broad blast radius (auth bypass, RCE, mass data exposure).
2. **No reproducer** — "looks vulnerable" without a PoC. The fix path is unverifiable.
3. **Generic findings** — "passwords aren't strong enough" without specifying policy + how to test. Be concrete.
4. **Missing context** — "endpoint X is BOLA" without showing how attacker discovers + exploits. Real attackers fuzz; assume they will.
5. **Compliance language without correctness** — "violates SOC2 CC6.1" without explaining the underlying weakness. Compliance is downstream of correctness; lead with the technical issue.
6. **Defense-in-depth dismissal** — declaring an issue "low severity because mitigated by WAF". Mitigations fail; defense at the source matters too.
